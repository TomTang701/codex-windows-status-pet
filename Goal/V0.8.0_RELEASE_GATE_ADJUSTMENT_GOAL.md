# v0.8.0 Release Gate Adjustment Goal

## Status

Supplemental Goal for the active v0.8.0 productization and release workflow.

This Goal does **not** restart v0.8.0 and does **not** invalidate evidence that has already passed.

Where this Goal conflicts with the existing v0.8.0 Goal, this Goal takes precedence **only for release-evidence and screenshot requirements described below**.

All other existing v0.8.0 scope, product behavior, engineering constraints, Git/GitHub safety rules, and release permissions remain unchanged.

---

## 1. Purpose

Adjust the remaining v0.8.0 release gates to match the environments that Codex and the maintainer can truthfully operate.

The previous release gate required separate Windows Sandbox, clean Windows 11 VM, or cross-Windows-user evidence. Those environments are not available to the current execution context and Codex cannot automate cross-user desktop interaction.

Do not build additional infrastructure solely to satisfy those unavailable environments.

The goal is to preserve strong release evidence while removing environment-specific blockers that do not materially improve v0.8.0 product correctness.

---

## 2. Scope

This supplemental Goal changes only the remaining v0.8.0 release evidence and documentation requirements.

### In scope

1. Reclassify unavailable clean Windows 11 VM, Windows Sandbox, and cross-Windows-user testing.
2. Keep real packaged-EXE lifecycle verification on the current physical Windows 11 x64 host.
3. Add or retain fresh GitHub-hosted Windows runner lifecycle evidence where practical.
4. Standardize authentic packaged-EXE screenshot capture.
5. Reduce the required screenshot set from eight to six non-duplicative screenshots.
6. Make the tray icon discoverable to end users by documenting its actual appearance and Windows notification-area location.
7. Continue the existing v0.8.0 verification and release workflow after the adjusted gates pass.

### Out of scope

- New product features.
- UI redesign.
- Tray icon redesign unless a real defect is discovered.
- New VM infrastructure.
- Windows Sandbox installation solely for this release.
- Cross-user desktop automation.
- Repeating already-passed test suites without a verification reason.
- Unrelated refactoring or cleanup.
- Expanding supported platforms beyond the existing Windows 11 x64 claim.

---

## 3. Release Gate Adjustment

### 3.1 Removed blocking requirements

The following are no longer blocking requirements for v0.8.0:

- A separate clean Windows 11 VM.
- Windows Sandbox evidence.
- A separate Windows user/profile controlled through cross-user desktop automation.
- Cross-Windows-user UI interaction evidence.

Record these truthfully as unavailable environment coverage or approved non-blocking limitations.

Do not report them as passed.

Do not emulate, simulate, or fabricate physical Windows 11 evidence.

### 3.2 Physical Windows 11 x64 host remains authoritative for client runtime evidence

Use the current physical Windows 11 x64 host and the real packaged v0.8.0 product artifact.

Verify the strongest practical lifecycle path supported by the current environment:

1. Install from the intended v0.8.0 release artifact or installer path.
2. Launch through the installed Start Menu entry.
3. Confirm the real packaged `CodexStatusPet.exe` runs.
4. Confirm normal tray/runtime behavior.
5. Exercise the supported reinstall or upgrade path.
6. Exercise normal uninstall.
7. Exercise `-PurgeSettings`.
8. Confirm unrelated `.codex` data is preserved.
9. Confirm the product does not require the repository checkout or development Python environment at runtime.

Use existing automated evidence when it already proves an item. Re-run only the checks required to establish fresh release evidence or to verify a changed path.

### 3.3 Fresh GitHub Windows runner evidence

Use a fresh GitHub-hosted Windows runner for package and install lifecycle verification where the existing workflow or a smallest-complete extension can do so safely.

The runner evidence may cover:

- Release ZIP/package contract.
- Installer or installation script execution.
- Install destination and expected installed files.
- Start Menu contract where observable.
- Reinstall or upgrade contract.
- Uninstall contract.
- `-PurgeSettings` behavior.
- Preservation of unrelated `.codex` data.
- Detection of repository, development venv, or source-tree dependency leakage.

GitHub-hosted Windows runner evidence is clean lifecycle automation evidence.

It must **not** be described as physical Windows 11 client evidence unless the actual runner operating system is Windows 11 and that fact is verified.

Do not create unnecessary CI architecture if existing release checks already provide equivalent fresh-runner evidence.

## 4. Authentic Packaged-EXE Screenshot Gate

The v0.8.0 screenshot requirement is reduced from eight screenshots to six screenshots.

All screenshots must come from the real packaged v0.8.0 `CodexStatusPet.exe`.

Source-run UI, mockups, reconstructed UI, AI-generated images, or development-only windows do not satisfy this gate.

### 4.1 Required screenshot set

#### English

1. Main overlay.
2. Main overlay context menu.
3. Settings window.

#### Simplified Chinese

4. Main overlay.
5. Main overlay context menu.
6. Settings window.

Total required: **6 authentic screenshots**.

A separate duplicate tray-icon screenshot is not required for each language.

### 4.2 Screenshot preparation state

Before capturing formal screenshots, normalize the product to the following state:

- Move the relevant application UI to the **primary display** when practical.
- `Window Size = 100%`.
- `Opacity = 100%`.
- Use the normal/default expanded layout.
- Ensure all five rows and the battery are fully visible.
- Ensure no content is clipped.
- Ensure the screenshot shows the real packaged v0.8.0 process.

The primary-display requirement exists because the current screenshot automation may fail to enumerate or capture the overlay correctly while it is on a secondary display.

A secondary-display screenshot automation limitation must not be treated as a product UI failure unless the application itself is shown to render incorrectly.

If automation still cannot enumerate the floating main overlay on the primary display, record the limitation and use authentic maintainer-provided screenshots from the packaged EXE for the affected view.

Do not lower visual authenticity requirements merely to satisfy automation.

## 5. Tray Icon Discoverability

Users must be able to identify the installed application's notification-area icon.

Inspect the **actual v0.8.0 packaged EXE runtime tray icon or authoritative icon asset** before writing documentation.

Do not guess the icon's appearance from the product name.

Update the closest authoritative user-facing documentation, normally the README or existing usage documentation, to explain:

1. What the actual CodexStatusPet tray icon looks like.
2. That it appears in the Windows notification area/system tray near the clock.
3. That Windows may place it under the `^` hidden-icons menu.
4. That the user can right-click the icon to open the product tray menu and access Settings.
5. Any important distinction between the CodexStatusPet tray icon and other nearby Codex or system icons, if the real icon makes confusion likely.

Prefer showing the real icon image or an authentic cropped screenshot when the repository documentation structure supports it.

Any image added to documentation must come from the real product asset or authentic runtime evidence.

Do not use a generated replacement icon for documentation.

1. the suite is needed for fresh release evidence,
2. a code or packaging change invalidated the prior result, or

Do not spend release time repeating a large passing suite only because this supplemental Goal was added.

## 7. Failure and Stop Rules

Follow the repository and global AGENTS.md debugging and verification rules.

For screenshot automation:

- The previously identified secondary-display enumeration limitation is known.
- First normalize to the primary display and the screenshot preparation state.
- Do not stack speculative UI automation fixes.
- If the packaged overlay remains non-enumerable, record the tool limitation and request only the specific authentic screenshot evidence that cannot be automated.

For release verification:

- Fix real product, packaging, installer, uninstall, privacy, or lifecycle defects.
- Do not change production behavior solely to make a screenshot tool recognize a floating window.
- Do not claim unavailable environment coverage as passed.
- Do not add VM infrastructure without Tom's explicit approval.

   - English context menu.

# v0.4.1 Correctness Stabilization Implementation Plan

> **For agentic workers:** Execute sequentially in the current session. Tom's allowed-skill policy does not authorize additional execution skills by default. Use `test-driven-development` for every production behavior change.

**Goal:** Eliminate minimum-scale reset-row clipping and route tray/quota transport failures through the authoritative five-row presentation path.

**Architecture:** Keep `derive_window_metrics` as the sole expanded metric source, preserve canonical geometry, and use measured content-safe vertical padding at 80% and 95%. Extend the existing pure status snapshot and presentation controller rather than adding a new subsystem. Tk integration tests exercise mapped production widgets at all 25 scale steps and actual queue paths; the UI applies all results with `StatusRows.configure_rows`.

**Tech Stack:** Python 3.12, `unittest`, Tkinter, existing pure API/controller/UI modules.

## Global Constraints

- Supported window scales remain exactly 80–200% in 5% steps.
- Expected expanded geometry remains 264×110, 330×138, 495×207, and 660×276 at 80/100/150/200%; vertical padding is 7 px at 80% and 8 px at 95%.
- Preserve five row IDs: `activity`, `progress`, `primary_5h`, `weekly`, `reset_credit`.
- No new module, dependency, configuration field/schema, network/provider, worker, process, polling loop, telemetry, feature, or broad refactor.
- Raw exception text must not be rendered.
- Workers enqueue only; all Tk updates remain on the main thread.
- No remote Git/GitHub action is authorized by this plan.

# v0.4.2 Autonomous Verification Conversion Implementation Plan

> **For agentic workers:** Implement task-by-task with the allowed `superpowers:test-driven-development` workflow; the repository does not authorize additional Superpowers execution skills.

**Goal:** Make every machine-observable release fact authoritative, uniquely classified, and executable through one formal RC command without routine human visual confirmation.

**Architecture:** Add one declarative verification inventory and validate it through Quality. Consolidate formal release execution in the existing RC runner, separate passes/blockers/limitations in machine-readable output, and make Windows subprocess decoding deterministic. CI calls RC once; existing focused tests and probes remain the authorities instead of creating a second host framework.

**Tech Stack:** Python 3.11+, `unittest`, JSON, GitHub Actions YAML, Markdown governance, Windows/Tk/Win32 existing tests.

## Global Constraints

- No product runtime behavior, configuration schema, provider, network path, dependency, polling, telemetry, or credential-boundary change.
- English normative documents remain canonical; registered Chinese pairs are synchronized translations.
- Simulation never becomes physical evidence.
- Quality remains non-release approval; RC remains the only formal automated release approval command.
- One fact has one authoritative check and one clear failure message.
- Existing v0.4.1 behavior and all 152 tests remain protected.

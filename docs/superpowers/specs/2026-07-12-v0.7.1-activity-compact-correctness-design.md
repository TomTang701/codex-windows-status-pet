# v0.7.1 activity localization and Compact geometry design

## Approved outcome

v0.7.1 fixes three focused v0.7.0 correctness defects without changing quota,
battery, row identity, tray, Shell, or persistence authorities.

## Root causes confirmed

- `activity_api` emits Chinese display strings, and `status_snapshot_api` passes
  those strings through in every language.
- `apply_settings()` always writes expanded root geometry but only repacks
  expanded content when not compact, leaving compact content in an expanded root.
- `drag()` validates a compact visible coordinate with expanded dimensions and
  persists that visible coordinate as the canonical expanded coordinate.

## Activity boundary

`snapshot_activity()` returns only language-independent semantic keys and
numeric active count. The activity contract uses `activity_state` for `idle`,
`thinking`, `tool_call`, `command`, `output`, and `completed`; it uses
`progress_state` for `no_active_conversations`, `active_conversations`,
`recent_conversation_completed`, and `session_directory_unreadable`.

`build_status_snapshot()` is the sole owner that converts those keys through
the existing localization API. It formats the active count into visible text
only after choosing the requested companion language. No reverse translation
or Codex UI-language inference is permitted.

## Compact geometry boundary

`settings["x"]` and `settings["y"]` remain canonical expanded-window
coordinates. A single main-window helper applies geometry for the current
manual mode after every settings application: expanded mode uses canonical
dimensions and coordinates; compact mode derives its square and visible
coordinates from the same canonical values.

The focused compact geometry module retains `compact_geometry()` and adds one
pure inverse helper that turns a legal compact visible coordinate back into a
canonical expanded coordinate. It preserves normal, right-edge, bottom-edge,
and bottom-right anchoring within integer-pixel tolerance.

Compact drag follows this flow:

```text
pointer -> desired compact visible position -> compact-size validation
-> displayed compact position -> canonical expanded conversion -> persistence
```

The drag path never expands the window and never validates the compact square
using expanded dimensions.

## Design verification

The design removes localized text from the activity source, localizes only at
the presentation boundary, preserves canonical expanded persistence, and keeps
manual Compact geometry authoritative through settings re-application and drag.
It changes no protected data authority or UI identity. **PASSED.**

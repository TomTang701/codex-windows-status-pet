# v0.6.1 quota-window identity design

## Decision

Use the official local app-server window's allowlisted `windowDurationMins` field as the sole v0.6.1 quota-window identity authority. Normalize a 300-minute window to the existing internal `rateLimits.primary` (5-hour) slot and a 10080-minute window to the existing `rateLimits.secondary` (weekly) slot. Do not use raw map key, array order, `primary`, or `secondary` names as business identity.

This choice is based on the recorded safe live evidence: the current app-server response has `rateLimits.primary.windowDurationMins = 10080` and `secondary = null`; the existing parser removes this metadata and causes the weekly window to render as 5-hour data.

## Alternatives considered

1. Keep raw keys as identity: smallest diff, but disproven by live weekly-only `primary` evidence.
2. Swap raw primary and secondary: fixes one screenshot only and remains positional speculation.
3. Classify allowlisted duration metadata: a focused parser correction that preserves current presentation/state owners. **Selected.**

## Data flow

```text
official account/rateLimits/read
→ quota_parse_api allowlists usedPercent, resetsAt, windowDurationMins
→ duration classification (300 → primary / 5h; 10080 → secondary / weekly)
→ existing QuotaState last-good boundary
→ status_snapshot_api rows and weekly battery presentation
→ shared expanded / compact BatteryView
```

Unknown durations are unclassified and do not populate either quota row or battery. Missing, null, malformed, or out-of-range data remains unavailable. Unknown and credential-like fields remain dropped at the parser boundary.

## Presentation semantics

- Classified 5-hour data renders only on `primary_5h`; missing 5-hour data renders `5h -- / --`.
- Classified weekly data renders only on `weekly`; missing weekly data renders `周 -- / --`.
- v0.6.1's fixed battery source is the classified weekly window. Weekly unavailable means battery unavailable, even when 5-hour data exists; no fallback is allowed.
- Existing stale/last-good snapshots retain their already classified data, so rows and battery stay coherent.

## Scope and ownership

- `scripts/api/quota_parse_api.py`: normalize safe duration metadata and classify windows.
- `scripts/api/status_snapshot_api.py`: consume the classified weekly window for the battery and preserve five rows.
- Existing transport, `QuotaState`, `BatteryView`, `main_window`, settings, and geometry owners remain unchanged unless a focused regression proves a necessary interface adjustment.
- No new quota/settings/layout framework, dependency, token access, telemetry, or sixth row.

## Design verification

This design uses safe official evidence, preserves absence semantics, prevents cross-window row leakage and battery fallback, and confines the correction to the parser/presentation boundary. It satisfies the v0.6.1 Design Verification question: **PASSED**.

## Required RED/GREEN evidence

- weekly-only raw `primary` with `10080` maps only to weekly;
- 5-hour-only raw window with `300` maps only to 5-hour;
- both windows, absent windows, unknown duration, malformed metadata/value, and sensitive-field drop;
- 5h and weekly row isolation;
- weekly battery selection and no fallback;
- stale/last-good coherence.

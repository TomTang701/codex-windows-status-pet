# v0.6.3 battery quota-source selector design

## Approved outcome

Add one persisted `battery_quota_source` setting with exactly two legal values: `primary_5h` and `weekly`. Its default, including a missing, malformed, null, numeric, or unknown value, is `weekly`. It selects only the quota window supplied to the existing pure `battery_presentation()` transformation; it never changes text-row visibility, quota parsing, or the selected source automatically.

## Control comparison and decision

Real Tk requested geometry for the control alone, using the current dialog padding and Chinese labels, was:

1. **Selected — discrete horizontal `tk.Scale`**: 252×70 with `from_=0`, `to=1`, `resolution=1`, no displayed numeric value, and explicit `5小时`/`每周` endpoint labels. It directly conveys left/right binary selection and prohibits an intermediate state.
2. **Two-button frame**: 210×54. It is smaller but reads as two commands rather than a persisted slider-like state.
3. **Two radio buttons**: 217×51. It expresses exclusivity but not the approved left/right slider affordance.

The selected Scale remains small relative to the existing settings dialog. It introduces no dependency, custom renderer, generic settings-control framework, or third state.

## Single data flow

`normalize_settings()` owns accepted values and defaulting. `SettingsSession` remains the transactional owner. `build_status_snapshot()` receives the normalized source, chooses only `rateLimits.primary` for `primary_5h` or only `rateLimits.secondary` for `weekly`, and passes that one result to the existing `battery_presentation()` function. `BatteryView`, compact mode, expanded mode, and the dialog do not compute remaining percentage, segment count, fallback, or source semantics.

If the selected window is absent, malformed, or unavailable, the battery is unavailable. It must never read the other window as a fallback. When the selected real window returns, the same selected source becomes available again without changing the setting.

## Dialog and lifecycle behavior

Apply updates battery presentation immediately without Save or restart. Save persists the source through the existing atomic configuration path. Close after an unsaved Apply restores the opening source and presentation. Restore Defaults sets source to `weekly` and keeps all v0.6.2 row flags true. Hiding a text row neither changes the source nor affects its battery data; changing source neither changes text-row visibility nor forces a row visible.

The dialog remains non-resizable, topmost, work-area placed, and owner-visible after close. Existing BatteryView interactions remain unchanged because no new overlay surface is introduced.

## Verification contract

- Config default/invalid normalization, persistence, backup, and settings transaction cover the two source values.
- Pure presentation tests prove selected 5h/weekly values, no fallback, unavailable selected source, recovery, 0/9/80/100 percent segment semantics, and one shared expanded/compact presentation result.
- UI tests prove endpoint mapping, Apply/Save/Close/Defaults behavior, and source/row-visibility independence.
- Both sources run against all eight row-visibility combinations, 25 scales from 80–200, DPI 96/120, compact/expanded content fit, interaction/lifecycle, mixed-DPI persistence, and Shell identity regressions.

## Design verification

The design has one configuration authority, one source-selection boundary, one existing battery transformation, two legal control values, no fallback, no duplicate semantics, and no change to the five-row or compact contracts. **PASSED.**

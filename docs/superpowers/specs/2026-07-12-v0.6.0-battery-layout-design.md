# v0.6.0 Approved Battery Visual and Layout Design

## Approval and Design Verification

Tom's approved battery-visual supplement supersedes the rejected ASCII battery and paw-retaining draft. **DESIGN VERIFICATION: PASSED.** The product visual is fixed; this document records the smallest implementation boundary and measured layout contract.

## Fixed product contract

- Expanded mode is five existing text rows on the left and one battery on the right; the paw is removed.
- The battery is a visual companion to `primary_5h`, not a sixth status row.
- It has exactly ten visible cells arranged as two columns by five rows.
- Segment order is bottom-left (1), bottom-right (2), then bottom-up row by row; a partially filled pair lights left before right.
- Compact mode is battery only. It shows the same complete 2×5/ten-cell structure, not a paw, emoji, percentage string, rectangle, or reduced cell count.
- Expanded and compact consume the same pure battery-presentation result.

## Truthful semantics

The existing official `rateLimits.primary.usedPercent` is converted by the existing presentation into remaining quota. The shared authority is:

```text
remaining = clamp(100 - official usedPercent, 0, 100)
remaining == 0 ? 0 : ceil(remaining / 10) = lit segments
```

Text and battery both represent remaining quota. Thus 9% → 1, 71% → 8, 78% → 8, 91% → 10. Rounding and flooring are forbidden. Missing/malformed primary quota yields unavailable battery state, visibly distinct from known 0%; stale/last-good quota renders the same last-good remaining value shown by text.

Fixed position bands are `1–2 red`, `3–4 orange`, `5–6 yellow`, `7–8 light green/yellow-green`, and `9–10 strong pure green`. The top band uses a distinct stronger-green color value from the 7–8 band.

## Implementation boundary comparison

| Boundary | Finding | Decision |
| --- | --- | --- |
| Focused `tk.Frame` plus ten child label cells | Native 2×5 grid, deterministic visibility/colors, no Canvas lifecycle, and cells can receive existing bindings. | Selected. |
| `tk.Canvas` plus rectangles | Precise but adds redraw/item lifecycle and Canvas interaction concerns. | Rejected. |
| Glyph/label text cells | Font/glyph geometry is not a reliable visual-cell contract across DPI. | Rejected. |

The battery is a focused `BatteryView` Frame owned by `main_window.py` or an equally focused UI adapter. No generic renderer framework, manager, theme engine, dependency, worker, or animation is justified.

## Measured layout contract

The old paw/ASCII measurement is rejected. A non-persistent Tk prototype with paw removed, five rows left, and a right-side two-by-five Frame battery measured at 100%/96 DPI:

```text
root actual/requested = 330×138 / 198×131
text actual/requested = 266×122 / 134×115
battery actual/requested = 40×115 / 40×115
five row heights = 23 each
```

This proves the approved structure fits without shrinking canonical window dimensions. It does not authorize arbitrary padding changes. The implementation must derive battery cell dimensions from `WindowMetrics`, then use all-scale/DPI requested-versus-actual tests to select any final small layout tightening. `BASE_WINDOW_WIDTH` and `BASE_WINDOW_HEIGHT` stay unchanged unless a measured RED proves otherwise.

## Interaction and regression contract

The battery Frame and every visible cell receive the existing right-click, enter/leave, drag start/motion/release bindings; no dead interaction surface is allowed. Removing the paw requires moving its bindings rather than dropping them.

Tests must cover pure remaining/ceil/color contracts; ten-cell 2×5 expanded and compact structure; paw absence; all 25 scales at DPI 96/120; unavailable/stale semantics; five rows; interaction/lifecycle; mixed-DPI persistence; and Shell identity.

## Self-review

No placeholder remains. The design has one semantic authority, one selected implementation boundary, and no contradiction between expanded and compact behavior. Scope is limited to the approved battery replacement and measured layout work.

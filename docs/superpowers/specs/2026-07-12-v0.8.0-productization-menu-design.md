# v0.8.0 Windows Productization and Menu Unification Design

## Approved outcome

Build a Windows 11 x64, PyInstaller **onedir** release package that installs
per user without Python, Git, pip, or a repository checkout. Preserve the
v0.7.1 application implementation and contracts, except that the tray and
overlay menus gain one shared product semantic model.

## Architecture

### Shared menu semantics

Add a Tk-independent menu model with stable action IDs, localization keys,
checked state, and a visibility action derived from live owner state. Both
`context_menu.py` and `tray_adapter.py` consume that model. The tray thread
only queues action IDs; `Pet.process_tray_actions()` remains the sole owner of
Tk calls, transitions, and persistence. Visible menus expose Hide; a hidden
overlay's tray menu exposes Show. Settings, topmost, lock, Compact, visibility,
and Exit occur in the same order and use existing translations.

### Release artifact

`scripts/build_release.py` is the only builder. On Windows x64 it verifies
version and pinned build dependencies, invokes committed
`packaging/CodexStatusPet.spec`, assembles one allowlisted runtime root, writes
the required manifest, notices, uninstaller, ZIP, and SHA-256 sidecar, then
rejects source/development material in the artifact. The executable uses the
existing `scripts/codex_status_pet.py` entry point and remains windowed.

### Install lifecycle

`install.ps1` validates an explicit ZIP plus SHA-256, validates the staged
manifest and entry point, stops only the installed app when necessary, replaces
only the per-user application directory through an app-owned backup, creates a
Start Menu shortcut, preserves settings, launches the installed EXE, and rolls
back a failed replacement. `uninstall.ps1` removes only app files and shortcut;
`-PurgeSettings` removes only this product's JSON settings file.

### Evidence and release boundary

Tests, Goals, documentation, CI, and source tools remain in the repository and
are denied from the user package. Static artifact, checksum, packaged-runtime,
menu, installer/upgrade/uninstall, and clean-environment checks are separate
authorities. README screenshots are captured only from the packaged executable:
English and Simplified Chinese each require tray, overlay, menu, and Settings.

## Alternatives considered

1. **Keep separate tray and Tk menus.** Lowest code change but cannot prove
   matching semantics or live check state. Rejected.
2. **Force both surfaces through Tk widgets.** Would violate pystray threading
   and make a hidden overlay unreachable. Rejected.
3. **Pure shared menu model plus surface adapters.** Minimal shared authority,
   preserves threading and localization boundaries. Selected.

For packaging, onefile was rejected because onedir provides inspectable runtime
dependencies and avoids per-launch extraction. An installer framework/MSIX was
rejected because the approved scope requires PowerShell, per-user installation,
and no new packaging framework.

## Design verification

The design preserves one production entry path, Tk main-thread ownership,
existing localization, settings, quota, activity, battery, Compact, DPI,
position, tray, mutex, and shutdown contracts. It has one authority for menu
semantics, one artifact builder, and a narrow installer boundary. It includes
the required authentic screenshot and clean-environment evidence without
claiming Sandbox/VM results as physical hardware evidence. **PASSED.**

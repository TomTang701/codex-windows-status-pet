# v0.5.3 Windows Shell Identity Correctness Design

## Outcome

The Codex Windows Status Pet remains a visible desktop overlay and tray companion while its main overlay window is absent from Win+Tab / Task View, Alt+Tab, and the ordinary taskbar application-button surface.

## Product identity

```text
Desktop overlay: visible
Tray icon: visible
Win+Tab / Task View: absent
Alt+Tab: absent
Ordinary taskbar application button: absent
```

This identity remains stable across settings, lock, Hide/Show, Compact/Expand, scaling, and supported DPI reapplication.

## Evidence and investigation boundary

Tom reports that Task View exclusion previously worked and has regressed.

Current source still uses `overrideredirect(True)`, so Tk borderlessness is not accepted as proof of Windows Shell exclusion.

Released v0.5.1 changed startup mapping order to `withdraw → target-position/DPI work → deiconify`. This is an investigation direction only. Production changes require the real HWND's style, extended style, owner/parent, visibility, and lifecycle state to be measured and compared with a known-working historical state.

## Failure path

```text
Tk root creation / mapping lifecycle
→ actual Win32 overlay HWND
→ effective style / extended-style / owner identity
→ Windows Shell classification
→ taskbar / Alt+Tab / Task View presence
```

The design must identify the exact boundary where current behavior differs from the known-working contract.

## Observable contract

For the main overlay HWND:

- the overlay exists and is visible when not hidden;
- the tray remains reachable;
- the measured shell-exclusion identity is present;
- conflicting ordinary application-window identity is absent;
- the identity remains stable through the supported lifecycle matrix.

The exact style or owner assertions are evidence-derived. The design does not preselect `WS_EX_TOOLWINDOW`, ownership, or a lifecycle reapply hook before measurement.

## RED

The authoritative RED must inspect the effective Win32 identity of the current overlay HWND and fail for the same measured condition that allows ordinary shell exposure.

`overrideredirect == True` is insufficient.

The RED must be capable of blocking the current regression before a release claim.

## Considered minimum approaches

1. Explicit evidence-required extended-style identity on the actual Tk root HWND.
2. Correct owner relationship if owner loss/change is the measured shell-classification cause.
3. Reapply the proven shell identity at the exact Tk/Windows lifecycle boundary that resets it.

The minimum evidence-supported approach wins. Do not combine approaches speculatively.

## Task View verification

Use Win32 identity first, then safe UI Automation or app-identifying host automation when available. Do not use undocumented internal Shell COM dependencies. Do not persist unrelated desktop or application-preview content.

A single maintainer physical fact is admissible only if the repository Human Interaction Admission Gate proves Task View presence cannot be truthfully automated.

## Regression surface

- cold start;
- settings open/Close/Apply/Save/Defaults;
- lock/unlock;
- Hide/Show;
- Compact/Expand;
- repeated settings cycles;
- 80–200% scale reapplication;
- DPI 96 and automated DPI 120 path;
- v0.5.1 geometry/content-fit;
- tray, single instance, shutdown.

## Scope

Only shell identity correctness is in scope.

The v0.6.0 battery indicator, paw replacement, layout tightening, compact battery geometry, installer, updater, and unrelated cleanup are excluded.

## DESIGN VERIFICATION requirement

Production code changes are prohibited until current HWND evidence, working/current comparison, one root-cause hypothesis, an authoritative RED, and two or three minimum approaches are documented and the design is marked `DESIGN VERIFIED`.

# v0.4.1 Correctness Stabilization Design

## Outcome

v0.4.1 closes two released-product correctness gaps without adding a feature:

1. every approved status row, including the full reset-credit date/time row, fits at 80%, 100%, 150%, and 200%;
2. tray and quota transport failures render through the existing presentation boundary instead of calling Label-only options on the `StatusRows` frame.

## Problem evidence

### Status-row clipping

A real Tk probe instantiated the production `Pet`, applied the approved text `重置 5 次 / 18:40 7/12`, mapped the window, and measured actual/requested geometry:

| Scale | Window | Window requested | Status allocated/requested | Reset row allocated/requested | Final row bottom |
|---:|---:|---:|---:|---:|---:|
| 80% | 264×110 | 173×111 | 204×94 / 113×95 | 204×18 / 113×19 | 102 |
| 100% | 330×138 | 208×135 | 256×118 / 134×115 | 256×23 / 134×23 | 125 |
| 150% | 495×207 | 308×200 | 386×177 / 199×170 | 386×34 / 199×34 | 185 |
| 200% | 660×276 | 413×255 | 517×236 / 270×215 | 517×43 / 270×43 | 235 |

At 80%, scaled height is `round(138 × 0.8) = 110`, while Tk requests 111. The status container loses one pixel and the final row receives 18 of its requested 19 pixels. The current scale tests check geometry, font, wraplength, padding, identity, and lifecycle, but not actual/requested content fit.

### Legacy error rendering

Two production paths execute Label-style calls against a `StatusRows(tk.Frame)`:

```python
self.text.config(text="Codex\n托盘图标异常", fg="#fca5a5")
self.text.config(text="Codex\n" + payload["error"][:30], fg="#fca5a5")
```

Integration probes produced the following results:

- `tray_error`: one logged Tk exception, unchanged five rows, and `tray_restart_scheduled == False` because the exception aborts before restart scheduling;
- quota transport error: one logged Tk exception, unchanged five rows, quota domain state changed to `unavailable`, and `last_error == "transport_error"`.

## Root-cause hypotheses

### Layout

Integer rounding is asymmetric with Tk font metrics at the minimum scale. Width remains exact at 264, but rounding 110.4 down to 110 allocates one pixel less than the measured requested content height. This is not a missing row, long-text wrapping problem, or monitor geometry problem.

### Error presentation

The v0.3.0 migration replaced one Label with a five-row Frame but left two emergency direct-string calls unchanged. `tk.Frame.config` does not accept Label options `text` or `fg`, so Tcl raises before the intended visible update and, for tray failure, before restart scheduling.

## Considered approaches

### A. Conservative metric rounding plus existing presentation path — selected

- Derive expanded height with `math.ceil(BASE_WINDOW_HEIGHT * scale)`; all other scale metrics remain unchanged.
- Add a mapped-Tk content-fit contract at all four approved scale points.
- Teach the existing status snapshot/controller boundary to produce approved failure rows.
- Apply every visible result through `StatusRows.configure_rows`.

This fixes the measured one-pixel deficit at its pure metric source, preserves deterministic scaling, adds no module/dependency, and consolidates rather than duplicates presentation.

### B. Reduce minimum-scale vertical padding — rejected

Reducing padding can make requested content fit, but it changes spacing semantics to compensate for a height-rounding loss and makes the smallest scale visually tighter. It treats the symptom at a child-layout parameter rather than the allocation source.

### C. Dynamically resize from Tk requested geometry — rejected

Querying mapped widget dimensions and mutating geometry at runtime would create a second size source, add geometry-loop risk, and weaken the pure canonical scale contract. The measured deficit is deterministic and does not justify runtime adaptation.

### D. Full runtime-state redesign — rejected

A new notice/state framework could model every failure, but v0.4.1 needs only two known paths. Phase 4 is explicitly conditional; introducing a broad state abstraction now violates scope and the new-abstraction budget.

## Observable contract

### Content fit

For each scale in `(80, 100, 150, 200)`, after mapping the production window with approved five-row text:

- root actual width/height equal `derive_window_metrics(scale)`;
- `root.winfo_reqheight() <= root.winfo_height()`;
- `StatusRows.winfo_reqheight() <= StatusRows.winfo_height()`;
- every label has positive actual height and `winfo_reqheight() <= winfo_height()`;
- every label's bottom is within the `StatusRows` actual height;
- the `reset_credit` label's right edge and bottom are within the status container;
- its requested width is no greater than actual width and its requested height equals one line's requested height;
- Compact/Expand and Hide/Show restore the same expanded metrics and retain all five row identities.

At 80%, expected expanded geometry becomes 264×111. At 100/150/200%, expected geometry remains 330×138, 495×207, and 660×276.

### Error presentation

- A tray error produces stable rows with activity `Codex`, progress `托盘图标异常`, approved error color `#fca5a5`, and schedules exactly one tray restart.
- A quota transport error with no last-good snapshot produces stable rows with activity determined by current activity, progress `额度暂不可用`, unavailable quota rows, and approved error color; raw exception text is not displayed.
- A quota transport error with last-good data retains the existing `QuotaState` truthfulness rule (`ok` inside freshness threshold, `stale` after it) and does not replace quota data with invented values.
- Normal, stale, unavailable, tray-error, and transport-error results all cross `StatusPresentationController` and reach Tk through `StatusRows.configure_rows`.
- No `self.text.config(text=..., fg=...)` path remains.

## Failure-path map

```text
window_scale_percent
→ derive_window_metrics
→ Pet.geometry and pack metrics
→ mapped StatusRows allocation
→ five Label allocations
→ complete single-line reset-credit row

tray_error queue action
→ Pet.process_tray_actions on Tk main thread
→ StatusPresentationController tray-error result
→ StatusRows.configure_rows
→ approved visible rows
→ should_schedule_restart

quota worker exception
→ production result queue
→ Pet.poll on Tk main thread
→ QuotaState.fail
→ StatusPresentationController normal/error-state render
→ StatusRows.configure_rows
→ truthful unavailable/stale result
```

Workers continue to enqueue data only and never call Tk.

## Interfaces and ownership

### `scripts/api/window_scale_api.py`

`derive_window_metrics(value) -> WindowMetrics` remains the only expanded metric source. Only height rounding changes from nearest integer to conservative ceiling. No new configuration field or schema change is introduced.

### `scripts/api/status_snapshot_api.py`

`build_status_snapshot(...)` remains the normal quota/activity mapper and adds explicit approved progress/color mapping for unavailable transport state without exposing exception text.

Add a narrowly bounded pure function:

```python
build_tray_error_snapshot(font_color="#fca5a5") -> dict
```

It returns the same presentation shape as `build_status_snapshot`: `text`, `rows`, `color`, `active_count`, `quota_tier`, and `quota_state`. Rows remain exactly the five stable identities.

### `scripts/api/status_presentation_controller_api.py`

Add:

```python
render_tray_error() -> dict
```

It delegates to the pure snapshot helper. The controller remains the single presentation boundary; no new manager/controller/module is introduced.

### `scripts/ui/main_window.py`

- `process_tray_actions` obtains the controller tray-error result, calls `configure_rows`, then evaluates restart scheduling. A rendering failure must still be logged by the existing outer handler; normal success reaches scheduling.
- quota transport error updates `QuotaState`, preserves last-good data when present, calls `render_status`, and continues. It never displays raw exception text.

## RED definitions

1. A mapped production `Pet` at 80% with approved rows fails because root requested height is 111 while actual is 110 and reset row requested height 19 exceeds actual 18.
2. A queued `tray_error` fails because one exception is logged, row values remain unchanged, and restart is not scheduled.
3. A current-generation queued quota transport error fails because one exception is logged and row values remain unchanged even though domain state becomes unavailable.

Each check directly reproduces the observed production path rather than importing only a helper.

## Regression surface

- Pure scale metrics and legacy scale migration.
- Mapped five-row content at 80/100/150/200%.
- Apply/Save/Close/Restore Defaults and config protection.
- Compact/Expand and Hide/Show at current scale.
- Normal, stale, unavailable, tray error/recovery, transport error with/without last-good data.
- Refresh generation/single-flight behavior and shutdown with scheduled callbacks.
- Context menu, settings dialog, tray actions, and package launcher.

## Security, privacy, and resource impact

- No network, provider, token, auth, telemetry, backend, process, worker, polling, dependency, or configuration change.
- Raw transport exception text is no longer rendered, reducing accidental diagnostic-data exposure.
- One `ceil` operation replaces one `round`; presentation mapping remains constant-time and main-thread-only.

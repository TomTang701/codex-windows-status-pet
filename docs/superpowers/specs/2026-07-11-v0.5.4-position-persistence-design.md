# v0.5.4 Position Persistence Correctness Design

## Outcome

A legal stable root position chosen by dragging the Codex Windows Status Pet survives mouse release, normal tray Exit, and the next launch unchanged unless display topology genuinely requires recovery.

The authoritative round-trip truth source is the actual stable root coordinate after the drag completes.

## Confirmed reproduction

Tom confirmed this production path:

```text
drag overlay
→ release mouse
→ tray menu Exit
→ launch again
→ overlay returns to the default/original position
```

Opacity and Window Size persist through the same normal Exit/restart sequence.

The authoritative reproduction therefore does not use Task Manager, forced kill, or Settings-dialog interaction.

## Position data flow

```text
actual Tk root after drag
→ drag()
→ settings["x"] / settings["y"]
→ finish_drag()
→ persisted JSON
→ normal tray Exit / close()
→ persisted JSON
→ Pet B load_settings()
→ safe_position()
→ recover_position()
→ startup geometry / apply_settings()
→ final Pet B root
```

The investigation identifies the first exact boundary where the intended post-drag coordinates stop being preserved.

## Round-trip truth source

After Pet A is dragged to a known legal position and Tk settles:

```text
expected_x = Pet A stable actual root x
expected_y = Pet A stable actual root y
```

The following must preserve `(expected_x, expected_y)`:

- runtime settings after drag;
- JSON after `finish_drag`;
- settings immediately before normal close;
- JSON after normal close;
- Pet B raw loaded settings;
- `safe_position` input;
- `safe_position` output;
- final stable Pet B root position.

For the authoritative RED, monitor topology is unchanged and the chosen position is known-valid. Recovery is not expected.

## Required trace

Capture at minimum:

```text
Pet A initial
after drag actual root
after drag settings
after finish_drag JSON
before close settings
after close JSON
Pet B raw loaded settings
Pet B before safe_position
Pet B after safe_position
Pet B final root
```

Also capture relevant window metrics, effective DPI, and monitor/work-area identity.

The root cause is the first divergence from the intended final root coordinates at an exact state, persistence, recovery, or startup-lifecycle boundary.

## Root-cause classes

### Runtime position ownership

Use only when the actual stable root is correct but runtime position state diverges before persistence.

Do not blindly synchronize `settings`, `hidden_position`, and `expanded_position` everywhere. Identify the authoritative owner and exact stale consumer.

### Persistence handoff

Use only when runtime state is correct but `finish_drag` JSON differs.

Investigate the exact dictionary and write boundary. Generic configuration persistence is not presumed broken because opacity and Window Size survive restart.

### Close-time overwrite

Use only when `finish_drag` JSON is correct and the normal close path subsequently replaces the coordinates.

### Startup recovery authority

Use only when normal-close JSON and Pet B raw load are correct but `safe_position` changes a legal coordinate under unchanged topology.

### Startup geometry/lifecycle reapplication

Use only when `safe_position` preserves the expected coordinate but the final root moves later.

## Settings Session boundary

Settings Session is secondary only.

The confirmed reproduction does not open Settings, so settings transactions are not part of the primary root-cause tree.

Settings-session position behavior is investigated only if the exact A-path cannot reproduce the issue or runtime evidence shows a settings session participates.

Settings Apply/Save/Close/Defaults remain in the final regression surface.

## Authoritative RED

The RED uses two production-equivalent `Pet` lifetimes sharing one temporary settings file:

```text
Pet A
→ drag to legal X,Y
→ settle
→ actual root = X,Y
→ finish_drag
→ JSON = X,Y
→ normal tray Exit
→ JSON = X,Y

Pet B
→ same settings path
→ raw loaded x/y = X,Y
→ safe_position preserves X,Y
→ final stable root = X,Y
```

The current released/current-main implementation must fail this contract before production changes.

A test that only checks `save_settings()` was called, only tests configuration normalization, or manually injects x/y without the real drag/close path is not authoritative.

## Candidate fixes

1. Correct the authoritative runtime position handoff when runtime state is the first divergence.
2. Correct persistence input/timing when the write boundary receives stale coordinates.
3. Correct recovery/startup position authority when saved/load coordinates are right but startup changes them.

Choose the smallest evidence-supported fix for the first divergence.

Do not combine fixes speculatively.

## Protected contracts

Preserve:

- v0.5.1 target-window-DPI geometry and five-row fit;
- v0.5.3 Shell Identity;
- `WS_EX_TOOLWINDOW=true`, `WS_EX_APPWINDOW=false` on the real root HWND where the current verified contract applies;
- legal multi-monitor coordinates;
- invalid/off-screen recovery;
- settings transaction semantics;
- opacity and Window Size persistence;
- Hide/Show and Compact/Expand;
- tray, single instance, and shutdown.

## Regression surface

- exact drag → finish_drag → JSON → normal Exit → JSON → new Pet → final root round trip;
- legal primary and secondary coordinates;
- negative-coordinate recovery contracts;
- invalid/off-screen recovery;
- locked drag does not move;
- Hide/Show;
- Compact/Expand;
- settings Open/Close/Apply/Save/Defaults;
- 80–200% scaling;
- DPI 96 and automated DPI 120 path;
- v0.5.1 geometry/content-fit tests;
- v0.5.3 real-HWND Shell Identity tests.

## Scope

Only position persistence correctness is in scope.

Excluded:

- battery indicator;
- paw replacement;
- layout tightening;
- compact battery geometry;
- installer/productization;
- auto-update;
- Shell Identity redesign;
- broad settings refactor;
- UI framework replacement.

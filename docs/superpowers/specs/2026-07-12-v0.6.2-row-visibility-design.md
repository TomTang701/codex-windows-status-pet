# v0.6.2 quota-row visibility and dynamic-distribution design

## Approved behavior

Add normalized `show_primary_5h`, `show_weekly`, and `show_reset_credit` booleans, all defaulting to `true`. Activity and progress are always present. A disabled optional row is removed from layout management; it is not replaced with empty text. An enabled row retains its actual unavailable or stale text. These preferences affect presentation only: quota parsing, the five stable row identities, battery source, text font, window metrics, and battery geometry retain their existing authority and behavior.

## Considered layout approaches

1. **Selected — repack the persistent labels with `pack(expand=True, fill="x")`.** `StatusRows` owns the five labels, calls `pack_forget()` for each, then repacks enabled labels in canonical `ROW_IDS` order. It adds no framework, coordinates, dependencies, or replacement widgets.
2. **`grid` rows with equal weights.** This can distribute rows, but it would introduce a second geometry-manager model inside an existing `pack`-based main-window composition. It offers no user-visible advantage over the selected approach.
3. **Absolute `place(y=...)` coordinates.** This would make row positions a second manual metric system and risks clipping or drift at 80–200% scale and DPI 96/120. It is rejected.

## Real Tk evidence

With the existing Segoe UI 10 labels in a 300×200 `StatusRows` region, a deiconified Tk root measured these doubled center coordinates:

- Five visible rows: `39, 119, 199, 279, 359`; each gap is `80`, i.e. 40 px.
- Weekly hidden: `49, 149, 249, 349`; each gap is `100`, i.e. 50 px.

The frame remained 300×200 in both cases. This proves that the selected approach reuses the same vertical text region and redistributes its visible labels evenly. The production window keeps its existing text/battery packing and measured `WindowMetrics`; no previous paw or ASCII-layout tuning is introduced.

## Runtime and settings flow

`config_api.normalize_settings()` owns defaulting and malformed-value normalization. `SettingsSession` remains the transaction owner. The dialog exposes the three booleans. Apply synchronizes the draft into the existing owner `Pet.apply_settings()`, which calls `StatusRows.set_visible_rows(settings)` before repacking the unchanged expanded content. Save persists that applied snapshot; Close restores the opening snapshot; Restore Defaults returns all three flags to true. Main-window rendering continues to write status values only through `configure_rows()`.

## Verification contract

- `StatusRows` preserves exactly five label objects and canonical visible order.
- Every one of the eight optional-row combinations has the required visible set; adjacent visible-row center gaps are equal within one physical pixel.
- At all 25 scale steps (80–200 by 5) and DPI 96/120, expanded mode shows each selected text row and the full 10-cell battery without overlap or clipping; window and battery metrics are unchanged by the settings combination.
- Compact mode remains battery-only and its ten cells stay visible.
- Apply, Save, Close, and Restore Defaults obey the existing settings-session transaction semantics, including reopening after an unsaved Apply.
- Existing interaction bindings, drag, hide/show, compact/expand, topmost, lock, mixed-DPI startup recovery, and Shell identity remain covered by targeted or established regression checks.

## Design verification

The design has no placeholder, contradictory quota authority, new row, hidden data substitution, or duplicate layout/semantics authority. It is limited to the approved v0.6.2 presentation settings. **PASSED.**

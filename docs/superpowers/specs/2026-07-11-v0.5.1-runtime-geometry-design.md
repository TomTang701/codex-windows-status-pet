# v0.5.1 Runtime Geometry Reapplication Stabilization Design

## Outcome

Cold start and every supported runtime settings, lock, and window-lifecycle transition preserve one coherent target-window-DPI expanded geometry, with all five stable rows completely visible.

## Root cause

The current `Pet` uses two DPI authorities:

- cold start derives runtime metrics before moving the HWND to the saved monitor;
- later `apply_settings` derives metrics from the positioned HWND;
- explicit point fonts remain affected by Tk's process-global initial scaling rather than the current window DPI.

This produces 330×138 at cold start and 264×110 after lock/settings reapplication for the same scale 80 on the same final DPI-96 monitor, while the text still requests 120-DPI heights.

## Considered fixes

### Preserve the cold-start/Tk-scaling layout DPI

Derive all pixel metrics from `tk scaling * 72`. This fits content but leaves a DPI-96 target window physically sized as DPI 120 and cannot truthfully satisfy target-monitor DPI transitions. Rejected.

### Change only Tk global scaling

Set `tk scaling` from `GetDpiForWindow` before each apply. The experiment remained clipped because existing point-font metrics were not rebuilt by this change. Rejected by runtime evidence.

### Target-monitor preposition plus explicit DPI-derived pixel fonts

Recommended. Keep the existing `Pet` and Window Scale API ownership:

1. create the root withdrawn;
2. load settings and derive provisional metrics only for safe-position recovery;
3. apply saved position only and call `update_idletasks` while withdrawn;
4. derive authoritative runtime metrics from the now-targeted HWND DPI;
5. create/configure text and paw fonts as explicit negative-pixel Tk sizes computed from logical point size and that same target DPI;
6. apply full expanded geometry from the same metric set;
7. every later `apply_settings` repeats steps 4–6 on the current target window;
8. `show_window` remains a visibility/position operation and does not own metric derivation.

No new module, controller, manager, persisted field, or layout subsystem is added.

## Interfaces

`Pet._sync_compatibility_metrics(settings)` remains the sole runtime metric authority and additionally stores the validated current layout DPI, for example `self.window_dpi`.

`Pet._font_spec(family, logical_point_size)` returns:

```python
(family, -max(1, round(logical_point_size * self.window_dpi / 72.0)))
```

Negative Tk font size means pixels, so geometry, padding, wrapping, and font pixels all consume the same target-window DPI without depending on process-global `tk scaling`.

## Initialization order

```text
Tk root creation
→ withdraw
→ load logical settings
→ provisional metrics for safe-position bounds only
→ recover saved position
→ position-only geometry while withdrawn
→ update_idletasks
→ GetDpiForWindow on target monitor
→ authoritative metrics + pixel font specs
→ create/pack widgets
→ full geometry
→ normal runtime show/mainloop
```

The provisional metrics must never become the final visible geometry unless the target DPI equals the provisional DPI.

## Runtime transition path

```text
settings/lock/restore event
→ apply_settings
→ current HWND target DPI
→ authoritative metrics + pixel font specs
→ full expanded geometry and packing
→ update/layout
→ five-row fit
```

Position-only `show_window` and `ensure_visible` preserve the most recently authoritative width/height.

## Observable contract

For one long-lived `Pet`, after every required transition:

- exactly five stable rows exist;
- every requested row height fits its allocation;
- every row and final row bottom remain inside actual client/root and status bounds;
- approved single-line rows fit without unexpected wrapping;
- unchanged logical scale and unchanged target DPI preserve identical expanded metric geometry;
- a changed target DPI produces the exact Window Scale API metrics for that DPI and matching pixel fonts;
- logical compatibility width/height remain 96-DPI values and do not persist physical pixels.

```text
same Pet cold-start fit = PASS (330×138)
→ toggle_locked

→ final row becomes 4/23

```

This check catches Tom's real symptom and must fail before production changes.

## Regression surface

- all required long-lived settings/lock/lifecycle transitions;
- all 25 scale steps at effective DPI 96 and 120 using long-lived instances;
- monitor-position recovery and legal secondary coordinates;
- logical settings persistence and transaction semantics;
- Hide/Show, Compact/Expand, tray/menu, single instance, and shutdown;
- v0.4.2 release gates and v0.5.0 lean-core boundaries.

## Failure handling

If Windows cannot return a positive DPI, retain the existing 96-DPI fallback. Pixel font size is clamped to at least one. No dynamic content-measurement padding or arbitrary geometry correction is permitted.

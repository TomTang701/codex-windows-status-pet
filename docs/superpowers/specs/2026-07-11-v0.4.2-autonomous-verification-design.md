# v0.4.2 Autonomous Verification Conversion Design

## Context and evidence

The v0.4.1 release proved that a green test can still be false evidence when it does not reproduce production initialization. Phase 2 therefore makes verification authority explicit and removes routine human confirmation for machine-observable facts.

Current duplication is concrete:

- CI runs Quality, package smoke, and whitespace separately even though formal RC runs the same gates.
- local release procedure commonly runs Quality/package/strict readiness before RC, then RC repeats them.
- the compatibility parser returns non-blocking facts under `deferred`, mixing classification and disposition.
- active testing/API prose still says UI changes require manual evidence even when Tk/Win32/process introspection can decide the fact.
- gate subprocesses use locale-default text decoding, so UTF-8 diagnostics can fail in the Windows parent before the actual child failure is reported.

## Considered approaches

### Documentation-only classification

Low implementation risk, but duplicate CI/local execution and locale-dependent output remain. Rejected because it does not meet the phase outcome.

### New comprehensive host automation framework

Could own launcher, tray, Tk, display, and process checks, but would duplicate existing tests and probes, add lifecycle risk, and violate the complexity replacement constraint. Rejected.

### Declarative inventory plus consolidation of existing gates

Recommended. Add one machine-readable inventory that points to existing authorities; validate uniqueness and classification; make RC the single formal orchestrator; update CI and process docs to call it once. This adds a small governance check while deleting duplicate procedure.

## Architecture

### Verification inventory

`docs/quality/verification-inventory.json` is the canonical classification dataset. Each item contains:

- stable `id`;
- one of `AUTOMATED`, `AUTOMATABLE`, `PHYSICAL-ONLY`, `OBSOLETE`, `DUPLICATE`;
- release `disposition`: `pass`, `blocker`, `limitation`, or `excluded`;
- one authoritative command, test module, evidence record, or replacement item;
- concise fact and rationale.

`scripts/check_verification_inventory.py` validates schema, unique IDs, allowed values, one authority, valid duplicate replacement IDs, and that physical-only unavailable facts use `limitation` rather than `pass`. Quality runs this checker once.

The JSON is language-neutral structured data. Its user-facing explanation is added to the existing bilingual Testing documents; no duplicate translated JSON is created.

### Gate orchestration

Quality remains the fast code-health composition and does not decide release readiness. RC remains the only formal release command and executes:

1. Quality once;
2. package smoke once;
3. strict compatibility readiness once;
4. whitespace once.

CI invokes RC instead of separately invoking its children. The package artifact remains uploaded from the RC-created `.build` output.

Both orchestrators decode child output explicitly as UTF-8 with replacement fallback. A malformed diagnostic cannot hide a child exit code.

RC emits one JSON result with:

- `release_candidate_approved`;
- `passes` keyed by successful authoritative gate;
- `blockers` keyed by failed gate with captured output;
- `limitations` copied from strict readiness;
- optional command detail for diagnosis.

No child gate runs twice inside one RC invocation.

### Compatibility disposition

`check_release_readiness.py` reports `passes`, `blockers`, and `limitations`. Historical physical passes remain passes; unavailable but explicitly unclaimed environments remain limitations. `Pending` or `partial` remains blocking unless explicitly marked non-blocking.

### Documentation conversion

Replace generic statements such as “UI changes require manual evidence” with:

- deterministic UI adapter/Tk/Win32/process evidence for machine-observable behavior;
- physical evidence only for unavailable topology/hardware or subjective appearance;
- one recorded limitation instead of repeated confirmation requests.

## Failure-path map

```text
inventory edit
→ inventory checker
→ Quality result
→ RC pass/blocker
→ CI status / local release decision

formal RC invocation
→ one child-gate execution each
→ UTF-8-safe captured output
→ passes + blockers + limitations
→ exit 0 only when blockers are empty

compatibility row
→ readiness classification
→ pass / blocker / limitation
→ RC summary
```

## Testing strategy

- Inventory schema, duplicate ID, invalid class/disposition, missing authority, and bad replacement tests.
- Readiness tests for pass/blocker/limitation separation and malformed status.
- RC tests proving each child executes once, limitations propagate, child failures block, and UTF-8 diagnostics survive.
- Workflow contract test proving CI calls RC and does not separately call Quality/package/whitespace.
- Full existing Quality, package smoke, strict readiness, and RC.

## Scope and compatibility

No product runtime file, configuration schema, dependency, network path, credential boundary, status text, menu, tray behavior, or window geometry changes. v0.4.2 is a release-engineering and verification-contract release only.

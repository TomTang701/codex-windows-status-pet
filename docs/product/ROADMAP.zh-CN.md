# Codex Windows 状态宠物更新路线图

**状态：** 活动路线图
**已发布基线：** `v0.5.0`
**当前唯一实现范围：** 无；产品化需要单独批准的设计

## 当前状态

- **已发布：** v0.3.2 至 v0.5.0 均已合并、打标签并正式发布；v0.5.0 删除了四个缺乏依据的生产边界。
- **已验证基线：** 合并后的 v0.5.0 通过 137 个核心测试和 22 个 Tk UI 测试；exact-head Windows CI 和合并后 RC 均通过，零阻塞并明确列出限制。
- **产品架构：** 外部伴侣使用本机官方 Codex app-server 数据、经过批准的本地会话元数据、五个稳定状态行、通知区域适配器，以及一个 80–200% 的规范“窗口大小”缩放值。
- **当前方向：** Phase 4 不需要实施。产品化是下一项最高价值机会，但在没有单独批准设计前不启动 v0.6.0 实现。
- **环境限制：** 当前没有混合 DPI 实体硬件、其他实体任务栏边缘或独立干净 Windows 机器，因此不声称这些实体证据。
- **明确排除：** 令牌读取器、第三方额度接口、遥测、托管服务，以及修改 Codex 核心或内置宠物文件。

## 产品目标

提供可靠的 Windows 伴侣：在任何显示器上均可访问，如实报告 Codex 活动与额度数据，设置可校验、持久化和恢复，并在实际可行时通过机器自动验证。

## 交付顺序

### Phase 0 — 仓库事实与基线

核对已发布状态，替换过期的活动治理文件，测量代码、测试、文档、依赖和门禁耗时，并对正确性、自动化和精简候选项排序。本阶段不改变运行时行为。

### Phase 1 — v0.4.1 正确性稳定

使用真实/请求 Tk 几何内容适配契约重现并解决最后一行裁剪。通过生产路径重现托盘和额度传输失败，并将可见输出统一路由到权威五行展示边界。不包含产品功能或大范围架构清理。

### Phase 2 — v0.4.2 自主验证转换

将每个 Quality、RC、兼容性和宿主检查分类为已自动化、可自动化、仅实体、过时或重复。每个机器可观察事实只保留一个权威检查，合并重复门禁逻辑，并一次性记录无法提供的实体证据，不再反复要求人工确认。

### Phase 3 — v0.5.0 精简核心

依据量化的消费者与兼容性证据删除死代码、移除没有受保护场景的兼容 API、合并透传包装层、统一重复的展示/状态路径，并减少活动流程文档中的重复内容。保留有意义的 Tk/Windows、本地传输、配置、解析、展示和刷新边界。

### Phase 4 — v0.5.1 条件式状态统一

仅当 Phase 3 后仍存在重复运行时或展示状态所有权时才执行；否则记录 `PHASE 4 NOT NEEDED`，不得人为制造工作。

### Phase 5 — v0.6.0 产品化决策

只有在精简核心和受支持宿主自动化得到证明后，才判断安装是否是最大的剩余易用性问题。安装器或启动项产品化需要单独验证设计，不会自动实施。

## 受保护方向

- Windows 11 x64 仍是受支持基线。
- 额度使用本机官方 Codex app-server；活动检测使用经过批准的本地会话元数据。
- 五个稳定行标识、单实例、Tk 主线程所有权、有界刷新、安全关闭、托盘可达性、事务式设置、按比例缩放和配置兼容性均受保护。
- 同一时间只允许一个活动版本和一个有边界的结果。
- 不进行大爆炸式重写，也不因实现细节测试仍在导入而保留死内部结构。

## 文档与验证规则

英文 manifest 文档是主版本；中文配对文件必须在同一变更中同步翻译。优先使用源码、契约测试、Tk/Win32/进程检查、安全的应用内交互和 GitHub 机器可读状态，而不是人工确认。量化证据写入带日期的质量/审计文件；易变 SHA 或一次性发布说明不写入长期路线图。

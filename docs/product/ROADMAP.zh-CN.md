- **已验证：** 74 项自动测试、manifest登记的17组中英文文档、Windows 11 启动器冒烟测试、副屏菜单/设置路径，以及键盘驱动的托盘隐藏/显示恢复。

- **应用模块化：** 右键菜单、设置窗口和托盘适配器已移入 `scripts/ui/`；Codex 发现/通信、安全诊断和状态展示已移入 `scripts/api/`；主窗口拆分仍按阶段推进。
- **待实体证据：** 混合 DPI 显示器、任务栏其他边缘、干净机器依赖安装，以及能实际触发收缩悬停展开的空闲桌面运行。
- **发布决策：** `scripts/check_release_readiness.py` 当前报告 6 个实体门槛阻塞项；在这些项目通过或得到明确批准前，v0.3.0 不标记为就绪。
- **明确不在范围：** 访问令牌读取器、第三方额度接口、遥测，以及修改 Codex 核心或内置宠物文件。

## 产品目标

构建一个可靠的 Windows companion：无论位于哪个显示器都能访问，准确显示 Codex 活动和额度数据，不臆造缺失值，并提供经过校验、可持久化且可恢复的设置。

## 交付顺序

### P0——可靠性与可访问性

1. 将右键菜单完整放入鼠标所在显示器的工作区，覆盖右下角、副屏幕、虚拟桌面负坐标和任务栏占用区域。
2. 集中校验坐标、窗口尺寸、颜色、字体大小、透明度、布尔值和刷新间隔。
3. 让刷新调度只能单实例执行、可取消，并且不阻塞 Tk 事件线程。
4. 保留现有安全的单实例互斥锁，绝不杀死无关进程。

**退出标准：** 无界面几何和输入校验测试通过；右键菜单第一次点击恰好执行一次；四个角都不会裁剪菜单；非法配置按字段回退。

### P1——设置与真实额度显示

1. 增加可持久化的宽度和高度。
2. 增加自由调整和按比例加减按钮。
3. 增加只允许数字、范围限制为 1–10 秒的刷新间隔设置。
4. 周额度和最近的未来重置额度到期时间使用本地 `HH:MM M/D` 格式显示，月份和日期不补前导零。
5. 保持保存、应用、恢复默认值和关闭的语义相互独立。

**退出标准：** 设置重启后仍然存在；应用不会关闭设置窗口；保存后持久化并关闭；关闭会丢弃草稿；日期格式有确定性测试。

### P2——借鉴 quota-float 的产品体验

1. 增加明确的加载中、过期、不可用、未登录和健康/警告/危险状态。
2. P0/P1 稳定后再考虑小圆图标和悬停展开模式。
3. 增加偏好设置备份/回滚和更强的窗口状态恢复。

当前仍以本地 app-server 数据源为默认。读取令牌或调用外部额度接口必须等独立安全评审和数据源契约批准后再实施。

### P3——文档与发布质量

英文文件是主版本；中文文件必须在同一提交中同步更新为翻译副本。每次重大或性能变更都必须更新 API 规范、增加针对性回归测试、更新变更日志并记录兼容性结果。`scripts/check_doc_parity.py` 会读取 `docs/document_manifest.json` 并检查登记文档对是否发生结构漂移。

## API 边界

| API | 职责 |
|---|---|
| `ConfigAPI` | 规范化、加载、原子保存和恢复设置。 |
| `InputValidationAPI` | 强制数字输入和类型范围。 |
| `SettingsSessionAPI` | 区分已保存、运行时、草稿和打开时设置快照。 |
| `DisplayGeometryAPI` | 枚举工作区并定位/限制弹出菜单矩形。 |
| `WindowSizeAPI` | 应用自由或等比例的宽高调整。 |
| `ResizeSessionAPI` | 基于基准尺寸应用可逆的百分比缩放。 |
| `QuotaFormatAPI` | 选择重置日期并格式化真实的本地文本。 |
| `QuotaStatusAPI` | 在无网络、无 UI 依赖的情况下分类额度健康状态。 |
| `DisplayModeAPI` | 决定是否启用空闲收缩并计算收缩尺寸。 |
| `CompactStateAPI` | 延迟空闲收缩，在活动/悬停时展开并保持边缘锚点。 |
| `WindowRecoveryAPI` | 只将真正离屏的窗口恢复到当前最近工作区。 |
| `TrayLifecycleAPI` | 校验托盘动作并保证恢复只调度一次。 |
| `RefreshSchedulerAPI` | 按有限的用户间隔保证 Quota 同时只有一次刷新。 |
| `RefreshControllerAPI` | 保证 Activity 与 Quota 刷新通道独立并安全处理 generation。 |
| `QuotaProviderAPI` | 将本地或未来数据源适配为统一快照结构。 |
| `QuotaParseAPI` | 只接受批准的额度字段和明确的命名别名。 |
| `QuotaStateAPI` | 跟踪 loading、最近成功、stale、signed-out 和 unavailable 额度状态。 |
| `ModelsAPI` | 在 UI 外部承载类型化额度领域值。 |
| `DiagnosticsAPI` | 持久化启动、工作线程、UI 和托盘错误。 |

任何性能功能都不得直接写在 UI 代码中，必须拥有独立 API 边界和兼容性测试。

## 兼容性矩阵

测试 Windows 10/11、单屏和双屏、负坐标和大坐标、100/125/150/200% DPI、任务栏四个方向、隐藏/显示、锁定/解锁、置顶/不置顶、损坏设置、刷新值 1/10/0/11/空值/非数字、网络超时、过期响应、缺失重置日期和重复启动。结果维护在 `docs/quality/COMPATIBILITY_MATRIX.zh-CN.md`。

## 文档规则

先更新英文主文件，再在同一提交中更新对应中文翻译；标题、API 名称、版本号、测试编号和表格结构必须保持一致。提交前运行 `python scripts/check_doc_parity.py`；检查结构一致性，但不要求中英文逐行相同。

## Git 与 GitHub 变更纪律

每次较大的、已经验证的变更都应及时提交并推送，不要把无关工作长期积累在工作区。较大变更包括新增 API、用户可见行为、设置结构、性能变化、安全边界、启动器变化或兼容性结果更新。

提交前必须：

1. 运行 `python scripts/run_release_checks.py`。
2. 运行 `git diff --check`。
3. 同步更新英文和中文规范/更新日志副本。
4. 检查暂存文件列表，并使用聚焦的提交说明。
5. 推送到已配置且经过确认的 GitHub 所有者；不得无声更换远程仓库或作者身份。

小型纯文档修正可以合并提交，但仍需通过文档一致性检查并及时提交。不得把无关的用户改动混入项目提交。

# v0.6.1 quota-window identity investigation

**Status:** Root cause established; production correction not yet implemented.

## Safe official local evidence

On 2026-07-12, the approved local `codex app-server --stdio` path was initialized and queried with `account/rateLimits/read`. The probe emitted only allowlisted quota identity/presentation facts; it did not read `auth.json`, tokens, prompts, arbitrary payload values, or unknown-field contents.

Observed safe facts:

- `rateLimits.limitId` was `codex`; `rateLimits.limitName` was null.
- `rateLimits.primary` was an object with `usedPercent`, `resetsAt`, and `windowDurationMins`.
- The actual `primary.windowDurationMins` was `10080`, which identifies a seven-day weekly window.
- `rateLimits.secondary` was null.
- The current response therefore represents a weekly-only quota state; no 5-hour window was present.

## Current parser boundary

`scripts/api/quota_parse_api.py` accepts only `usedPercent` and `resetsAt`, drops `windowDurationMins`, and assigns raw `primary` directly to the normalized `primary` slot. `scripts/api/status_snapshot_api.py` then presents that slot as `primary_5h` and uses it for the battery.

## Root-cause hypothesis

The companion's classification defect is caused by positional/raw-key identity assumptions combined with loss of the safe official `windowDurationMins` field. A weekly-only `primary` window is therefore rendered as 5-hour data, while the weekly row and the v0.6.0 battery do not consume the available weekly value.

## Required correction boundary

Normalize only the safe duration metadata required for classification, classify 300-minute windows as 5-hour and 10080-minute windows as weekly, retain absent windows as absent, and make the v0.6.1 battery consume the classified weekly window without fallback. Unknown or credential-like fields remain dropped.

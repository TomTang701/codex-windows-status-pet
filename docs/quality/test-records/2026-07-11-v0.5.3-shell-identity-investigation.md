# v0.5.3 Shell Identity Investigation

## Scope

This record investigates only the Windows shell identity regression in the released v0.5.1 overlay baseline. It does not change layout, battery, paws, installer behavior, or productization scope.

## Provenance

The inspected running production process used the bundled `pythonw.exe` and `scripts/codex_status_pet.py`, started after the v0.5.1 release. Its source baseline was the v0.5.1 code path plus v0.5.3 documentation-only activation; this rules out the earlier stale pre-v0.5.1 incident class for this investigation.

## Real root HWND evidence

The probe obtains the effective top-level window with `GetAncestor(widget_hwnd, GA_ROOT)`, not from Tk configuration alone.

| Evidence | Historical working `bd11bb0` | Current v0.5.1 behavior |
| --- | --- | --- |
| Root extended style | `0x00080088` | `0x00080008` |
| `WS_EX_TOOLWINDOW` | yes | no |
| `WS_EX_APPWINDOW` | no | no |
| Owner | `0` | `0` |
| Visible | yes | yes |
| Tk `overrideredirect` | yes | yes |

The first real-HWND RED test failed as expected: `overlay root HWND ... lost WS_EX_TOOLWINDOW`.

## Lifecycle timeline

The historical constructor applied alpha/topmost before the first native mapping. Final mapping produced `WS_EX_TOOLWINDOW`.

The v0.5.1 constructor calls `withdraw()`, applies early position geometry plus `update_idletasks()`, then runs `apply_settings()` with `-alpha`. The early native mapping first establishes the tool-window bit, but `-alpha` changes the extended style from the intermediate tool-window state to layered-only. The final `deiconify()` leaves `0x00080008` rather than restoring `WS_EX_TOOLWINDOW`.

## Minimal-fix comparison

| Candidate | Result | Decision |
| --- | --- | --- |
| A. Write the style before alpha / generic creation-time adjustment | Alpha can replace the bit before final mapping. | Rejected |
| B. Assign an owner window | Owner was `0` in both working and failing builds, so it does not explain the regression. | Rejected |
| C. Normalize the real root HWND immediately after `deiconify()` | Manual restoration yielded `0x00080088`; it survived apply settings, hide/show, compact/expand, lock changes, and alpha changes. | Selected for TDD implementation |

## Design Verification status

**DESIGN VERIFIED.** The defect is reproduced by a real root-HWND assertion, and the selected candidate has direct lifecycle evidence. Final release approval requires Quality/package/RC evidence, a fresh launcher provenance check, and desktop/tray smoke verification.

## Candidate verification

The v0.5.3 candidate turned the same focused real-HWND tests GREEN. The focused lifecycle test covers cold start, settings application with alpha/scale/topmost changes, lock, Hide/Show, and Compact/Expand. The full production-equivalent runtime probe additionally checks the real root style after cold start, settings open/close/Apply/Save/Defaults, repeated settings cycles, Hide/Show, Compact/Expand, and the combined sequence. Every sampled state retains `WS_EX_TOOLWINDOW=true` and `WS_EX_APPWINDOW=false`.

Fresh root-launcher evidence on 2026-07-11 started PID `24772` from the bundled `pythonw.exe` and current project script. Its visible root HWND had extended style `0x00080088`, `WS_EX_TOOLWINDOW=true`, `WS_EX_APPWINDOW=false`, and owner `0`. A second root-launcher invocation retained exactly one project `pythonw.exe` process. Windows automation's ordinary application inventory did not enumerate the overlay, consistent with the required non-ordinary Shell identity.

`scripts/run_quality_checks.py`, `scripts/package_smoke_test.py`, and `scripts/run_release_candidate_checks.py` each passed with zero blockers after the focused Green result. The strict RC recorded only pre-existing non-blocking environment limitations.
